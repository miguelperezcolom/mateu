# Why Mateu

Mateu is a strong fit when a team wants to build **business UIs** with less code, less duplication and fewer moving parts.

It is especially compelling in three scenarios:

- you need to build CRUD-heavy backoffice screens fast
- you want each microservice to own its own UI
- you want to embed business UIs into an existing frontend without rewriting everything

---

## 1. Fast admin panels

If your team needs:

- management consoles
- internal tools
- operations screens
- CRUD-heavy backoffice modules

Mateu lets you move extremely fast because the model drives the UI.

👉 [See the admin panel case →](/java-user-manual/use-cases/admin-panel)

---

## 2. Distributed backoffices

If your architecture is distributed, Mateu lets each service define its own UI and contribute it to a shared shell.

That means:

- less frontend coordination
- clearer ownership by bounded context
- no separate frontend integration layer

👉 [See the distributed backoffice case →](/java-user-manual/use-cases/distributed-backoffice)

---

## 3. Embedded UI

If you already have React, Vue or another frontend, Mateu does not force a rewrite.

You can expose a Mateu UI and embed it with a web component.

That means:

- lower adoption risk
- gradual rollout
- backend teams can own business modules directly

👉 [See the embedded UI case →](/java-user-manual/use-cases/embedded-ui)

---

## What Mateu is not trying to optimize for

Mateu is not the best fit when your main priority is:

- pixel-perfect consumer-facing UI
- highly custom animation-heavy frontend work
- deep frontend specialization as the center of the product

Its sweet spot is **business UI**, not handcrafted frontend artistry.

---

## The practical path

For most teams, the easiest learning path is:

1. build the quickstart
2. read the admin panel case
3. explore distributed backoffice or embedded UI based on your architecture
