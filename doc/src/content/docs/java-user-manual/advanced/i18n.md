Mateu pasa todas las cadenas visibles al usuario — etiquetas de campo, títulos de página, mensajes de validación, textos de botones, alertas — por un bean que implementa la interfaz `Translator`. Esto centraliza la traducción en un único punto de extensión.

---

## La interfaz `Translator`

```java
public interface Translator {
    String translate(String text, HttpRequest httpRequest);
}
```

| Parámetro | Descripción |
|---|---|
| `text` | Cadena original (tal como aparece en el código o en las anotaciones) |
| `httpRequest` | Contexto de la petición — permite leer la cabecera `Accept-Language`, cookies de sesión, o cualquier otro dato necesario para resolver el locale |

El valor devuelto es la cadena que se enviará al frontend.

---

## Implementación por defecto

El módulo `core` proporciona una implementación basada en **Java i18n estándar** (`ResourceBundle`). Para usarla basta con incluir un fichero de mensajes en el classpath:

```
src/main/resources/
  messages.properties          ← locale por defecto
  messages_es.properties       ← español
  messages_fr.properties       ← francés
  ...
```

Ejemplo de `messages_es.properties`:

```properties
Name=Nombre
Save=Guardar
Order\ #=Número de pedido
Field\ is\ required=El campo es obligatorio
```

El locale se resuelve a partir de la cabecera `Accept-Language` de la petición HTTP.

---

## Proporcionar una implementación alternativa

Para sustituir la implementación por defecto basta con registrar un bean que implemente `Translator` con mayor prioridad. En Spring Boot:

```java
@Component
@Primary
public class DatabaseTranslator implements Translator {

    final TranslationRepository repo;

    @Override
    public String translate(String text, HttpRequest httpRequest) {
        String locale = httpRequest.getHeaderValue("Accept-Language");
        return repo.findTranslation(text, locale)
                   .orElse(text);  // fallback al texto original
    }
}
```

En Quarkus o Micronaut usa la anotación equivalente (`@Alternative` + `@Priority`, `@Primary`, etc.).

---

## Casos de uso habituales

| Escenario | Implementación sugerida |
|---|---|
| Traducciones en ficheros `.properties` | Implementación por defecto de `core` |
| Traducciones en base de datos (editables en caliente) | Bean `@Primary` que consulta la BD |
| Etiquetas por tenant / marca blanca | Leer el tenant del request y resolver el bundle correspondiente |
| Servicio externo de traducción | Llamada a API con caché local para no impactar la latencia |
| Modo desarrollo sin traducciones | Devolver `text` directamente (pass-through) |

## Consideraciones de rendimiento

La implementación de `Translator` se invoca por cada cadena de cada respuesta. Si la resolución implica acceso a BD o red, añade una capa de caché keyed por `(text, locale)`:

```java
@Component
@Primary
public class CachedDatabaseTranslator implements Translator {

    final TranslationRepository repo;
    final Cache<String, String> cache = Caffeine.newBuilder().maximumSize(10_000).build();

    @Override
    public String translate(String text, HttpRequest httpRequest) {
        String locale = resolveLocale(httpRequest);
        return cache.get(text + "|" + locale,
                         k -> repo.findTranslation(text, locale).orElse(text));
    }
}
```
